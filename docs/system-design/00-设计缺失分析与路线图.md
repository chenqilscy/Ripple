## 一、诊断结论（一句话版）

> **三大模块只有"造浪池"被深度设计；底座（数据模型 / 权限 / 文件存储）和闭环两端（云霓 / 冰山）均缺失。** 若立即编码，将出现"中段流光溢彩、首尾无路可走"的尴尬。

## 四、与现有文档的"接缝"问题（已识别的不一致）

1. **Schema 不一致**：
   - 技术架构白皮书 §二.2 给出 `(:Node {id, content, embedding, type})`
   - AI Prompt 规范、协作文档、跨湖文档对 Node 属性又有各自描述
   - 需在 G1 中**收敛为唯一权威 Schema**，其他文档以"引用"方式回链

2. **删除语义不一致**：
   - 协作文档定义了"软删 + 迷雾区"
   - 状态机文档定义了 `Disabled` 状态
   - 白皮书定义了"蒸发"
   - 三者要在 G1 中统一为同一份生命周期模型

3. **权限词汇不一致**：
   - 白皮书：未提
   - 协作文档：舟子 / 渡客
   - 网关文档：JWT Claims（未定义角色枚举）
   - 需在 G1 中统一

4. **AI 触发时机不一致**：
   - Prompt 规范：异步后台
   - 协作文档：WebSocket 推送
   - 网关文档：同步 REST
   - 需在 G5 中统一为"事件驱动 + 推流"

5. **状态术语扩张未收敛**：
   - 故事 v2 新引入：凝露 / 迷雾区 / 雾气节点 / 漂流瓶 / 朦胧版 / 蓄能潭
   - 必须在 G1 节点生命周期模型中纳入：`Mist`(雾气) → `Drop`(浮萍) → `Frozen`(固形) → `Vapor`(蒸发/迷雾区) → `Erased`(物理删除)；以及 `Public Bottle`(漂流瓶) 的独立子类型
