## 一、协作模型

### 1.1 基础原则

- **所见即所得**：用户 A 的"投石"应即时出现在用户 B 的湖面上
- **最终一致性**：允许短暂的状态差异，但最终所有用户的湖泊状态必须一致

### 1.2 角色定义

详见 [G1 §五权限矩阵](G1-数据模型与权限设计.md)：OWNER / NAVIGATOR / PASSENGER / OBSERVER。
本文统一称呼："**舟子**"=OWNER，"**渡客**"=PASSENGER。

## 二、实时通信架构

### 2.1 技术选型

- **协议：** WebSocket（全双工）
- **后端：** FastAPI + WebSockets，或专用 Socket.IO
- **广播：** Pub/Sub（Redis Streams）

### 2.2 数据流

```

    │
    ▼
```

## 三、冲突解决

### 3.1 状态冲突（位置/属性）

**场景：** 用户 A 与 B 同时移动同一节点。
**方案：** **LWW (Last Writer Wins) + Vector Clocks**

- 每个操作携带时间戳和用户 ID
- 时间戳相同时按 User ID 字典序决定胜负（deterministic）
- 前端节点平滑过渡到胜出位置，不产生抖动

### 3.2 意图冲突（"暗流分岔"）

**场景：** 用户 A 想把 N 连到 X，用户 B 想把 N 连到 Y。
**方案：** **不阻止任何一方**

- A 的操作生成实线连接 X
- B 的操作生成虚线连接 Y
- 系统提示：**"此处暗流涌动，请选择主航道。"**
- 由舟子 (Owner) 决定保留哪条，或保留为"双向暗河"

### 3.3 删除冲突（软删 + 迷雾区）

**场景：** A 正在编辑节点内容，B 把节点蒸发了。
**方案：**
- B 执行蒸发：节点进入 VAPOR 状态（迷雾区可恢复，详见 [G1 §三](G1-数据模型与权限设计.md)）
- A 的编辑操作仍可提交
- 系统提示 A："**该节点已被蒸发，您的修改将存入迷雾区，是否恢复？**"

## 四、前端反馈：同舟者（Co-presence）

### 4.1 光标与化身

- 每个用户分配独特颜色（A=青碧，B=日出橙）
- 光标旁显示头像或昵称（"舟子-阿树"）
- 拖动节点时留下短暂的"水痕"特效（几秒后消散）

### 4.2 权限视觉化

- 他人节点：默认半透明
- 锁定状态（他人正在编辑）：节点边缘旋转的光环（水涡），提示"此处正在施工"

---

## 五、后端数据同步

### 5.1 操作转换 (OT)

- 把"移动节点"视为对 (x, y) 属性的 `add(delta)` 操作
- 服务器转换并发操作，确保最终位置收敛

### 5.2 Neo4j 事务

- 所有 WS 触发的数据库操作必须通过 Neo4j Transaction（原子性）
- 例：创建节点 + 建立 `[:BELONGS_TO]` 关系必须在同一事务内
