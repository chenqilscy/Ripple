# M4 需求清单（草案）

> 状态：草案 · 2026-04-23 起草 · 待 PM/接口设计师/正方/反方四方评审后定版

## 立项背景

M3 已交付：流式编织 SSE / 凝结 Crystallize / 推荐器骨架 / Spaces 协作 / 性能基线（万 QPS / 列表 2k+ QPS）。

M4 进入"实时协作 + 多模态 + 移动端"三方向并行，目标：让 Ripple 从单人灵感工具升级为团队创意工作台。

## 三大方向

### 方向 A · 实时协作冲突解决（CRDT / OT 二选一）

**痛点**：
- 当前 WS 仅广播事件，多人同时编辑同一节点会出现"后写覆盖"。
- 节点 content 字段没有版本向量，无法识别冲突。

**目标**：
- 多人同时编辑文本节点，能合并而非覆盖。
- 离线编辑可在重新上线后自动合并（最终一致）。

**候选方案**：
1. Yjs（CRDT，社区成熟）+ y-websocket 网关
2. Automerge（CRDT，更易调试）
3. 自研 OT（轻量，但维护成本高）

**关键问题**：
- 后端选 Go，Yjs 主要是 JS 生态；需要桥接（y-websocket 服务独立部署，或 yrs Rust binding via FFI）
- 节点 schema 需引入 doc 字段，旧数据迁移策略

### 方向 B · 多模态（Image / Audio）

**痛点**：当前节点仅支持文本，灵感往往以草图/语音形式出现。

**目标**：
- 节点支持上传图片（最多 5 张，单图 ≤ 5MB）
- 节点支持录音（≤ 60s，自动转写为文字附在 content）
- AI 编织/凝结能识别图片关键元素（用 vision 模型）

**关键问题**：
- 对象存储：MinIO 已规划，未上线；M4 必须落地
- vision provider：智谱 GLM-4V / OpenAI gpt-4o / 本地 Florence2 三选一
- 鉴权：图片/音频的下载链需要短期签名 URL（防热链）

### 方向 C · 移动端（PWA + React Native 二选一）

**痛点**：当前仅 Web 桌面端，灵感在路上常无法记录。

**目标**：
- iOS / Android 至少一端可用
- 离线能新建节点（待联网同步）
- 推送：邀请加入空间 / @ 提及触发系统通知

**候选方案**：
1. PWA（最快，复用现有 React 代码；推送限制大）
2. React Native（双端原生，开发成本翻倍）
3. Flutter（重写整个前端，pass）

**关键问题**：
- 离线同步与方向 A 的 CRDT 强相关
- 推送通道：Web Push（PWA） vs APNs/FCM（RN）
