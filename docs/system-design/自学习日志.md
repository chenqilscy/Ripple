## 2026-04-22 · 第 5 轮审查发现 Condense 越权漏洞

**触发**：autopilot 执行完成后，反方第 5 轮代码审查发现高危越权问题。

**反馈**：
- 原 Condense 仅校验“目标 lake 写权限”，未校验“源节点归属与权限”。
- 攻击路径：用户 C 在自己 lake-c 拥有 OWNER，可构造 {node_id: 他人 MIST, lake_id: lake-c} 将他人节点挪入自己的湖。

**解读**：
- 状态迁移接口必须做双边鉴权：源端权限 + 目标端权限。
- 仅做目标湖校验会造成跨湖节点劫持。

**决策**：
- 修复 service.NodeService.Condense，强制同时校验：
  1) 节点 owner 或源 lake 写权限
  2) 目标 lake 写权限

**行动**：
- ✅ 修复 service/node.go:Condense 源端权限校验
- ✅ 新增 TestNode_Condense_PreventsCrossLakeHijack 覆盖越权路径
- ✅ 修复 PGCallRecorder.Close 与 Record 并发关闭导致的 send on closed channel panic
- ✅ 补充 service/node.go:Condense 注释，明确 EDITOR / PASSENGER 角色语义

**预防**：
- 所有“跨资源状态迁移”接口必须执行双边鉴权 checklist：源端权限 + 目标端权限 + 状态迁移合法性。
- 异步 worker 的 Close/Record 并发路径必须遵循：先关标志位，再在写入路径加锁或短路，禁止向已关闭 channel 写入。

## 2026-04-27 · Phase 13 staging 端口冲突复盘

**触发**：实际回收后重新 bootstrap staging，远端 `fn.cky` 存在共享 Redis/Postgres 与历史前端测试容器，默认端口 `16379`、`15432`、`14173` 发生冲突。

**反馈**：用户要求继续第三轮深度代码审查并修复；本轮审查将真实演练暴露的问题转为配置修复。

**解读**：staging compose 不能假设联调机端口完全空闲；脚本也不能把端口写死到 healthcheck/smoke 输出，否则端口覆盖后会误报失败。

**决策**：保留默认端口兼容旧流程，同时让 backend / yjs / frontend / PG / Redis 等 staging 端口都支持环境变量覆盖；文档明确端口冲突时的覆盖方式。

**行动**：更新 `docker-compose.staging.yml`、`scripts/bootstrap-staging.ps1`、联调手册、真实联调清单、快速上手与 Phase 13 状态文档；新增 rebootstrap smoke 验收记录。

**预防**：后续新增 compose 端口必须使用 `STAGING_*_PORT` 环境变量形式，并在 bootstrap 脚本中同步读取，避免配置与脚本分叉。
