## 2026-04-29 · AI 任务错误信息最小披露

**触发**：Phase 15 C.1+/C.2+ 第 6 轮 GPT 审查发现 `AiJobWorker` 将底层 `err.Error()` 写入 `ai_jobs.error`，`ai_status` 又向湖内 Observer 原样返回，可能泄露 LLM provider、数据库、网络或仓储内部细节。

**反馈**：任务失败原因不能把内部错误原样返回给普通成员；湖内可读状态应只暴露可行动、脱敏后的错误文案，详细原因保留在服务端日志。

**解读**：AI Workflow 的异步任务状态属于用户可见 API，错误字段同样受“最小披露”约束；日志可保留根因，但持久化任务错误和 API 响应必须脱敏。

**决策**：worker 只写入固定的用户可见错误分类；`ai_status` 对历史任务错误统一二次脱敏；新增单测防止内部错误重新泄漏。

**行动**：
- ✅ `backend-go/internal/service/ai_job_worker.go` 改为写入固定错误分类；
- ✅ `backend-go/internal/api/http/handlers_ai_trigger.go` 增加 `sanitizeAIJobError()`；
- ✅ `backend-go/internal/api/http/handlers_ai_trigger_test.go` 增加内部错误脱敏回归测试。

**预防**：任何异步任务表的 `error` 字段默认按用户可见字段处理；底层异常只能进日志，不直接进入 API 响应。

## 2026-04-28 · Prometheus metric 名称陷阱 + Grafana dashboard PromQL 修正

**触发**：Round 5 T42 监控仪表板，dashboard JSON 写入时使用了错误的 metric 名称，导致 Grafana panel 所有 PromQL 查不到数据。

**反馈**：代理发现 dashboard JSON 中含有 3 类错误：  
1. `ripple_http_request_duration_seconds_bucket` → 实际是 `ripple_http_request_duration_ms_bucket`（单位 ms，非 seconds）  
2. `ripple_db_pool_acquired` / `ripple_db_pool_total` → 实际是 `ripple_db_pool_acquired_conns` / `ripple_db_pool_total_conns`（有 `_conns` 后缀）  
3. PromQL `* 1000` 做毫秒转换 → 多余，metric 已是 ms，直接除会变 μs 量级  

**解读**：  
- 自研 metrics 系统（`internal/metrics/vars.go`）变量名不走 Prometheus 命名惯例（`_seconds` 后缀），直接用 `_ms`；写 dashboard 前必须 grep `vars.go` 确认实际名称。  
- 无 route/status label：`HTTPRequests` 和 `HTTPDuration` 均 label=nil，不支持 `by (route, method)` 或 `{status=~"5.."}` 过滤。  

**决策**：  
- dashboard 中 "5xx 错误率" panel 改为 "LLM 调用总数"（用已有 `ripple_llm_calls_total`）  
- "HTTP 请求速率（按路由）" panel 改为汇总速率（`rate(ripple_http_requests_total[1m])`）  
- 移除所有 `* 1000` ms 转换  

**行动**：  
- `monitoring/grafana/provisioning/dashboards/ripple-overview.json` 已修正 8 处 PromQL  
- 写入用户记忆 `ripple-go.md`：metric 名以 `_ms` 结尾（非 `_seconds`），DB pool 有 `_conns` 后缀  

**预防**：  
- **新规则**（写入本文档）：编写 Grafana/Prometheus PromQL 前，必须先 grep `backend-go/internal/metrics/vars.go` 确认 metric 名称和 label 列表，不得凭经验猜测。

## 2026-04-28 · Autopilot 第二/三轮经验沉淀（PS 数组 + cleanup 串入 + 编码）

**触发**：连续两轮 autopilot（去 GH Actions + Phase 14.6-B/C/D + 准入清单）执行过程中，反复踩到 PowerShell 脚本编码 / 数组 / 子脚本调用陷阱。

**反复问题模式**：

1. **`create_file` 不是覆盖**：在已存在文件上 `create_file` 是追加，导致脚本头被翻倍；唯一稳定的覆盖方式是 `Set-Content -NoNewline -Encoding UTF8` 或 `[System.IO.File]::WriteAllText(path, content, (New-Object System.Text.UTF8Encoding $true))`（强制 BOM）。
2. **PowerShell 5.1 ANSI 解析中文 .ps1**：无 BOM 的中文 .ps1 被 GBK 解释，引号配对失败，报「字符串缺少终止符」。**对策**：脚本英文化 + UTF-8 BOM；只允许注释含中文。
3. **PS 单元素管道结果不是数组**：`$failed = $results | Where-Object { ... }` 仅命中 1 个时返回标量，`$failed.Count` 为该对象的 `Count` 属性（通常 0），导致 ALL PASS 与 FAIL 同时出现。**对策**：`$failed = @(...)` 强制数组化。
4. **子脚本调用退出码**：用 `& powershell -File child.ps1` 调用子脚本时，必须用 `$LASTEXITCODE` 判断，而非 `$?`；内部 `throw` 会让 `$LASTEXITCODE=1`。
5. **后台 SSH 密码必须 send_to_terminal**：Windows 没有 sshpass / plink，OpenSSH 的密码 prompt 必须用 `send_to_terminal` 单次发送，不能批量。
6. **Staging 容器名假设**：默认 `ripple-staging-pg` 错，实际是 `ripple-staging-postgres`（compose service 名）；任何 `docker exec -i <container>` 调用前必须先 `docker ps --format '{{.Names}}'` 确认。

**预防措施（写入位置）**：

- 上述 1/2/3/4 已沉淀到用户记忆 `ripple-go.md`、`loadtest-harness.md`；
- 5/6 沉淀到 session memory（演练阶段）；
- 后续所有 staging 脚本默认值必须用 compose service 全名，且支持 `-PgContainer`/`-Neo4jContainer` 参数覆盖。

**行动**：
- ✅ `phase14-6-acceptance.ps1`：`$failed = @(...)` 数组化；新增 `-IncludeStagingCleanup` 串入子脚本（Commit `a88213f`）；
- ✅ `staging-cleanup-smoke.ps1`：默认 PG 容器名修正为 `ripple-staging-postgres`（Commit `2ac92f7`）；
- ✅ Settings E2E 增强 OWNER 撤销 confirm 高危流（Commit `d73c37d`）；
- ✅ Phase14 准入清单去 CI 化（Commit `76dfff2`）。

## 2026-04-29 · P18/P19-A/C 独立安全审查 + 前端 ref/render 分离 + 准入门修复

**触发**：会话完成 P19-C 协作光标后，老板选择"全部执行"扩展到 P19-A 和 P18 系列全模块安全审查。

**发现问题**：
1. P19-C `AiTriggerButton.tsx`：`pollCount.current` 在 JSX render 中被访问，违反 `react-hooks/refs` lint 规则；同时因为是 ref，UI 显示的轮询计数不会更新（功能 bug）。
2. vitest 误拾 `e2e/` Playwright 文件，导致 `npm run test` 报 2 个 FAIL（"Playwright Test did not expect test.describe() here"）。
3. P19-A：`/explore` 路由仅在 `d.LLMRouter != nil` 时挂载，staging 无 LLM key 时 404 是预期行为（不是 bug）。
4. P18-B：`baseURLFromRequest` 信任 `r.Host`（可被伪造）+ `clientIP` 不处理反向代理——此两项标注低优先级。

**解读**：
- `useRef` 值不触发 re-render，若 UI 需要展示 ref 值，必须同步维护一个 display state；不能直接把 ref 改成 state（会引入 stale closure，因为 `poll` callback 未加 `pollCount` 到依赖数组）。
- vitest glob 默认包含 `**/*.{spec,test}.ts`，`e2e/` 下的 Playwright `.spec.ts` 文件满足该 pattern；应在 `vite.config.ts` 的 `test.exclude` 中排除 `e2e/**`。
- 准入门（go vet / go test -race / lint / unit / build / E2E）每次 staging deploy 后都必须跑完整一轮。

**决策**：
- `pollCount` 保持 `useRef`（逻辑判断，避免 stale closure），新增 `pollCountDisplay` state 仅用于 render；在 `increment` 和 `reset` 时同步更新两者。
- `vite.config.ts` 新增 `test: { exclude: ['e2e/**', '**/node_modules/**'] }`。
- P18-B 两项问题标注为"技术债"，留后续迭代处理（低优先级，无立即利用风险）。

**行动**：
- ✅ commit 67d1951：AiTriggerButton pollCount ref/render 分离 + vitest 排除 e2e 目录。
- ✅ 全准入门通过：go vet / go test -race ./internal/... / lint / unit 4/4 / build / E2E 5/5。
- ✅ staging smoke: healthz + P18-C/D 限制验证 PASS。

**预防**：
- 任何 ref 值若同时用于 render，必须用 `ref + displayState` 双写模式，不得直接改 ref 为 state。
- 新增 E2E spec 文件后，检查 vitest exclude 列表是否覆盖对应目录。
- 跑准入门时，`npm run test` 的 2 failed 不代表测试失败——先看是否是 e2e spec 混入的误报。

## 2026-04-22 · 第 5 轮审查发现 Condense 越权漏洞

**触发**：autopilot 执行完成后，反方第 5 轮代码审查发现高危越权问题。

**反馈**：
- 原 Condense 仅校验“目标 lake 写权限”，未校验“源节点归属与权限”。

- 结论：**INSERT OK / READ OK（nodes=2, edges=1）/ LIST columns OK（6 cols）/ CLEANUP OK** — P21 DB smoke ALL PASSED。
- HTTP API 层（CreateSnapshot + GetSnapshot）通过代码审查 + 本地单测覆盖；staging HTTP 层暂跳过（账号门禁问题）。
