## 一、总体架构分层

```
┌─────────────────────────────────────────────┐
│  表现层  Three.js / WebGL  Delta View         │  ← D1, D2, D3
├─────────────────────────────────────────────┤
│  应用层  FastAPI + WebSocket                 │  ← D4 API 网关 / D5 协作
├─────────────────────────────────────────────┤
│  推理层  LLM + Embedding 织网引擎             │  ← G5, D7
├─────────────────────────────────────────────┤
│  数据层  Neo4j + PostgreSQL + S3              │  ← G1, G4
└─────────────────────────────────────────────┘
```

## 二、核心概念速览

| 概念 | 定义 | 详见 |
| :--- | :--- | :--- |
| Lake | 工作区，承载图结构 | G1 §二 |
| Node | 浮萍节点（5 种状态：MIST/DROP/FROZEN/VAPOR/ERASED） | G1 §三 |
| RELATES_TO | "暗流"连边 | G1 §2.2 |
| REFERENCES | 跨湖引用（运河模式） | D6 |
| Snapshot / Iceberg | 资产快照与策展 | G3 |
| MIST / 凝露 | 云霓雾气节点的入水仪式 | G2 |
| Outbox + Workflow | AI 触发的统一事件驱动 | G5 §七 |

## 三、湖泊命名体系

## 四、跨湖连通（流域思想）

| 模式 | 技术实质 | 隐喻 | 适用场景 |
| :--- | :--- | :--- | :--- |
| 运河（引用） | `[:REFERENCES]` 关系 | 京杭大运河 | 跨湖只读引用 |
| 合流（合并） | Sub-graph 迁移 | 江河入海 | 项目完结，归并入资产湖 |

详见 [D6 跨湖连通设计](D6-跨湖连通设计.md)。

## 五、智能织网引擎（多 Agent）

- **角色：** 织网者 / 探源者 / 评审员 / 沉淀者
- **机制：** 红蓝军对抗 + 最大 2 轮迭代 + 评审低分降级为虚线"潜在暗流"
- **触发：** Outbox 事件驱动，与业务事务解耦

详见 [G5 AI 服务编排](G5-AI服务编排-织网工作流.md)。

## 八、技术栈推荐

| 层 | 选型 |
| :--- | :--- |
| 前端 | React + Three.js + Zustand |
| 后端 | Python 3.12 + FastAPI + Pydantic v2 |
| 任务 | Celery + Redis Stream |
| 数据 | PostgreSQL 16 + Neo4j 5 + S3-compat |
| AI | OpenAI / Anthropic API + 本地 Embedding |

## 九、MVP 路线建议

阶段一（MVP）：单湖图结构 CRUD + 单次 AI 织网（无 Critic）+ 基础 Delta View
阶段二：多湖 + 跨湖引用 + 完整 AI 编排（含 Critic）+ 实时协作
阶段三：云霓移动端 + 冰山发布 + 公开分享
