# M2 路线图：边、协作、历史

> 阶段定位：M1 已完成创意单点闭环（湖 / 节点 / 造云 / 凝结 / 蒸发 / 实时），M2 进入**多人协作 + 知识结构化**。
> 本文档为设计提纲，不含落地代码。每个特性后附"实施前需明确的决策点"，待老板裁决后再细化。

---

## 总览

| 特性 | 价值 | 复杂度 | M2 优先级 |
|------|------|--------|-----------|
| **F1 边（节点连线）** | 让节点之间形成知识图谱，从"散点"到"网络" | 中 | P0 |
| **F2 PASSENGER 邀请** | 让朋友进湖体验创意流，扩大单湖参与人数 | 中 | P0 |
| **F3 节点编辑历史** | 防误删/找回旧版/支持时间机器视图 | 中 | P1 |
| **F4 多人光标 + 在线状态** | 协作存在感，类似 Figma 头像 | 低 | P2 |

依赖关系：F1 是 F3、F4 的基础（光标、历史都依赖节点已有 ID）。
F2 与 F1 解耦，可并行。

## F1 · 边（Edge）数据模型与 API

### 1.1 现状回顾
- 节点：PostgreSQL（权威）+ Neo4j（可选镜像，目前未启用边）
- M1 节点之间无显式关系；只有「凝结」会在父子节点（同 owner）形成隐式从属

### 1.2 需求
1. 用户能在两个节点之间画线，表示「关联 / 因果 / 派生 / 反对」等关系
2. 关系可命名（自由文本）或选枚举（`relates`/`derives`/`opposes`/`refines`/`groups`）
3. 删除节点时：**软删边**（标 deleted_at），不级联删另一端的节点
4. 实时广播：建/删边事件让其他在场用户看到

### 1.3 数据模型

**PostgreSQL · 权威**：
```sql
CREATE TABLE edges (
  id            TEXT PRIMARY KEY,             -- 业务侧 ULID
  lake_id       TEXT NOT NULL REFERENCES lakes(id) ON DELETE CASCADE,
  src_node_id   TEXT NOT NULL REFERENCES nodes(id),
  dst_node_id   TEXT NOT NULL REFERENCES nodes(id),
  kind          TEXT NOT NULL,                -- relates/derives/opposes/refines/groups/custom
  label         TEXT NOT NULL DEFAULT '',     -- kind=custom 时的自由文本
  owner_id      TEXT NOT NULL REFERENCES users(id),
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at    TIMESTAMPTZ,                  -- 软删
  CONSTRAINT no_self_loop CHECK (src_node_id <> dst_node_id),
  CONSTRAINT same_lake CHECK (TRUE)            -- 应用层校验：src/dst 必须同 lake
);

CREATE UNIQUE INDEX idx_edges_unique_alive
  ON edges(src_node_id, dst_node_id, kind)
  WHERE deleted_at IS NULL;

CREATE INDEX idx_edges_lake_alive ON edges(lake_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_edges_src ON edges(src_node_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_edges_dst ON edges(dst_node_id) WHERE deleted_at IS NULL;
```

**Neo4j · 投影**（可视化 + 路径查询用）：
```cypher
// 与 PG 通过 outbox 异步同步（与 M1 节点同套机制）
CREATE (n1:Node {id: $src})-[r:EDGE {id: $id, kind: $kind, label: $label}]->(n2:Node {id: $dst})
```
**软删 Neo4j 边**：直接 DELETE（Neo4j 仅作投影，权威在 PG）。

### 1.4 API 草案

```
POST   /api/v1/edges                  body: {src_node_id, dst_node_id, kind, label?}
GET    /api/v1/lakes/{id}/edges       query: include_deleted=false
DELETE /api/v1/edges/{id}             软删
```

权限规则：
- 创建：actor 必须对 src 和 dst 节点所在 lake 有写权限
- 删除：edge.owner OR lake 写权限

### 1.5 实时事件

| Type | Payload |
|------|---------|
| `edge.created` | `{edge_id, src, dst, kind, label, owner_id}` |
| `edge.deleted` | `{edge_id}` |

### 1.6 决策点（待老板裁决）

1. **边是否分方向**？目前模型按有向（src→dst）。无向场景（如"相关"）也用有向 + UI 不显示箭头；还是另加 `directed:bool` 字段？
2. **kind 是否锁死枚举**？建议：枚举 + custom 二选一。custom 需 label 非空。
3. **跨 lake 引用**？M2 暂不支持（CHECK same_lake）。
4. **MIST 节点能否连边**？建议：可以——边也参与"凝结"。即如果两端都凝结后才永久；任一端蒸发则边自动软删。

## F2 · PASSENGER 邀请流程

### 2.1 现状
- 角色已定义：OWNER / EDITOR / PASSENGER / VIEWER（详见系统约束规约）
- 但只有 OWNER 创建湖；其他角色加入流程未实现

### 2.2 需求
1. 湖 OWNER 能生成「邀请链接」，附可选过期时间和角色
2. 受邀人点击 → 登录/注册 → 自动入湖
3. OWNER 可吊销未消费的邀请
4. PASSENGER 入湖：可读全部节点 + 创建自己 owner 的节点 + 不能改/删别人的

### 2.3 数据模型

```sql
CREATE TABLE lake_invites (
  id              TEXT PRIMARY KEY,
  lake_id         TEXT NOT NULL REFERENCES lakes(id) ON DELETE CASCADE,
  token           TEXT NOT NULL UNIQUE,         -- 32 字节随机 hex（URL safe）
  role            TEXT NOT NULL,                -- EDITOR/PASSENGER/VIEWER
  inviter_id      TEXT NOT NULL REFERENCES users(id),
  invitee_email   TEXT,                         -- 可选：定向邀请
  expires_at      TIMESTAMPTZ,                  -- NULL = 永不过期
  max_uses        INT NOT NULL DEFAULT 1,       -- 0 = 不限
  used_count      INT NOT NULL DEFAULT 0,
  revoked_at      TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_lake_invites_token ON lake_invites(token) WHERE revoked_at IS NULL;
CREATE INDEX idx_lake_invites_lake ON lake_invites(lake_id);

-- M1 已有的 lake_members 表用来记录已加入用户：
-- lake_id, user_id, role, joined_at（M1 应已实现，否则 M2 补）
```

### 2.4 API 草案

```
POST   /api/v1/lakes/{id}/invites       body: {role, expires_in?, max_uses?, invitee_email?}
                                        resp: {invite_url: ".../join?t=xxx", token, expires_at}
GET    /api/v1/lakes/{id}/invites       列出未过期未吊销的
DELETE /api/v1/lake-invites/{id}        吊销（OWNER）

POST   /api/v1/invites/accept           body: {token}
                                        效果：当前登录用户写入 lake_members
GET    /api/v1/invites/preview          query: token=xxx
                                        无需登录预览：返回 lake.title + inviter.name
                                        （供"未登录看到这个湖叫什么"）
```

### 2.5 风险

- **泄露风险**：token 出现在浏览器历史/截图。缓解：
  - max_uses 默认 1
  - expires_in 默认 7d
  - 邀请使用后立即失效
- **暴力枚举**：32 字节 hex 已足够熵（2^128），无需额外限流
- **身份冒认**：invitee_email 仅作提示，不强制比对（避免邮箱未验证场景阻塞）

### 2.6 决策点

1. **是否需要邮箱通知**？M2 建议**不做**，URL 复制粘贴足够。M3 再接 SMTP。
2. **是否允许"游客只读"匿名访问**？建议**不允许**——所有访问必须登录，符合数据隔离底线。
3. **邀请链接是否需要 OWNER 二次审批**？建议**不需要**——OWNER 生成时已决定信任。

## F3 · 节点编辑历史

### 3.1 现状
- 节点 content 直接 UPDATE，旧版本丢失
- 只有 deleted_at 软删

### 3.2 需求
1. 每次节点 content/title 变更 → 留版本快照
2. UI 提供"时间轴"，可回滚到任意历史版本
3. 历史保留策略：永久 OR 保留近 30 天 + 100 版本（防爆）

### 3.3 数据模型

**事件溯源式**（推荐）：
```sql
CREATE TABLE node_revisions (
  id          TEXT PRIMARY KEY,
  node_id     TEXT NOT NULL REFERENCES nodes(id) ON DELETE CASCADE,
  rev_number  INT NOT NULL,                    -- 单调递增（节点内）
  content     TEXT NOT NULL,                   -- 完整快照（不做 diff，简化）
  title       TEXT NOT NULL DEFAULT '',
  editor_id   TEXT NOT NULL REFERENCES users(id),
  edit_reason TEXT NOT NULL DEFAULT '',        -- 可选注释
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(node_id, rev_number)
);
CREATE INDEX idx_node_revisions_node_time ON node_revisions(node_id, created_at DESC);
```

写入时机：
- 节点 Create → 写入 rev_number=1
- 节点 Update → 在事务里同时 INSERT revision

回滚：
- 「回滚到 rev N」= UPDATE node SET content=rev_N.content + INSERT 新 rev (rev_N+1) 标 edit_reason="rollback to N"

### 3.4 API 草案

```
GET  /api/v1/nodes/{id}/revisions          query: limit=50
GET  /api/v1/nodes/{id}/revisions/{rev}    单版本详情
POST /api/v1/nodes/{id}/rollback           body: {target_rev_number}
```

### 3.5 决策点

1. **保留策略**：永久 vs 30天/100版本？
   - 永久：磁盘成本可控（创意节点 content 一般 < 1KB）
   - 建议：M2 永久，M3 评估后再加 GC
2. **是否支持 diff 视图**？M2 建议**不做**——前端展示完整快照对比即可。
3. **凝结节点 (PERMA) 编辑是否仍记历史**？建议：是。

## F4 · 多人光标 + 在线状态（次要）

### 4.1 现状
- WS 已建立 lake topic 订阅
- 无在线用户列表 / 光标位置广播

### 4.2 设计
- 每个 ws 客户端 join 时记录 `lake_presence:{lake_id}` Redis SET（成员= user_id）
- 客户端每 5s 发 `cursor.update` 帧 → broker 广播到同 topic
- **不持久化**——离线即丢

### 4.3 决策点
- M2 是否值得做？建议：**P2，可砍**。优先 F1+F2+F3。

## 跨 M2 的非功能需求

### NFR-1 性能
- 边查询：单 lake 边数 < 1000 时 listEdges 应 < 50ms
- 历史回溯：单节点 revision 查询应分页且 < 100ms

### NFR-2 数据完整性
- 所有写操作走 Outbox saga（M1 已建立）
- 边创建必须校验 src/dst 同 lake（应用层 + DB CHECK 双保险）

### NFR-3 安全
- 邀请 token 必须用 `crypto/rand` 生成，永不复用
- API 全部走现有 AuthMiddleware（含 query token 兜底）
