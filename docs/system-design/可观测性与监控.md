# G6 · 可观测性与监控设计

**版本：** v1.0
**日期：** 2026‑04‑21
**适用对象：** SRE / 后端 / AI 运维
**核心隐喻：** 水文监测站

> 没有度量，就没有改进。本文档规定「青萍」全链路可观测性方案。

---

## 一、设计原则

1. **三大支柱齐全**：Logs / Metrics / Traces，共享 `trace_id` 关联
2. **AI 调用必须可追溯**：每次 LLM 调用包含 prompt、token、cost、latency、Critic 评分
3. **用户可感知**：关键指标驱动前端降级提示与限流文案
4. **隐私优先**：节点 content 不进入 metrics/logs；仅哈希或长度

---

## 二、技术栈

| 组件 | 选型 | 用途 |
| :--- | :--- | :--- |
| 协议 | OpenTelemetry (OTel) | 跨服务统一标准 |
| Metrics 后端 | Prometheus + Grafana | 时序与看板 |
| Logs 后端 | Loki + Grafana | 结构化日志 |
| Traces 后端 | Tempo / Jaeger | 全链路 Trace |
| 告警 | Alertmanager + 飞书机器人 | 阈值告警 |

## 五、Traces 标准

每个请求注入：
- `trace_id`（全链路）
- `user_id` / `lake_id`（业务上下文）
- 跨服务通过 `traceparent` Header 传递

AI 调用 Span 必须包含属性：
```
ai.workflow=weave_relation
ai.agent=critic
ai.model=gpt-4o
ai.tokens.in=1234
ai.tokens.out=89
ai.score=0.78
ai.tier=premium|standard|fallback
```

## 六、Dashboard 设计

### 6.1 「水文气象台」总览（Grafana）
- 当前在线舟子数 / 活跃湖泊数
- 投石速率（节点创建/min）
- 决堤口流量（发布/h）
- API P99 延迟趋势

### 6.2 「AI 织网车间」
- LLM 调用次数 / 模型分布
- Token 消耗与成本（按湖、按用户）
- Critic 评分分布
- 降级链触发计数

### 6.3 「迷雾区健康度」
- VAPOR 节点总量
- TTL 清理积压
- 误删恢复操作次数

## 七、告警规则（节选）

| 告警 | 阈值 | 严重度 | 用户文案 |
| :--- | :--- | :--- | :--- |
| API 5xx 比例 > 1% | 持续 5min | P1 | "潮汐异常..." |
| AI 错误率 > 50% | 持续 3min | P1 | "AI 暂歇" + 触发降级 |
| Token 单湖小时成本超预算 | > 上限 | P2 | 限流该湖 |
| Outbox lag > 60s | — | P1 | — |
| Neo4j 连接池耗尽 | > 90% | P1 | — |

---

## 八、日志规范

- **格式：** JSON
- **必含字段：** `timestamp / level / service / trace_id / user_id / lake_id / event / msg`
- **禁止字段：** 节点 `content`、`prompt` 原文（只记录 hash + 长度）
- **保留期：** 30 天热存，1 年冷存
