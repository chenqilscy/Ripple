# 协作冲突解决方案 · Phase 3-P1-01

> 文档状态：草案，待下一轮实现
> 版本：v0.1 | 2026-04-30

---

## 1. 问题描述

多用户同时编辑同一节点 content 时，当前实现为 **Last-Write-Wins**（最后写入覆盖）：

```
用户A 读取 content="hello"      → 修改为 "hello world"
用户B 读取 content="hello"      → 修改为 "hello neo4j"
用户A 先写入 → content="hello world"
用户B 后写入 → content="hello neo4j"  ← 用户A 的修改丢失
```

---

## 2. 方案对比

| 方案 | 实现复杂度 | 数据完整性 | 适用场景 |
|------|-----------|-----------|---------|
| Last-Write-Wins（现状） | 无 | 丢数据 | 不可接受 |
| 乐观锁 CAS（版本号） | 低 | 冲突提示 | MVP ✅ |
| CRDT（无冲突复制数据类型） | 中 | 自动合并（字符级） | M4+ |
| OT（操作变换） | 高 | 自动合并（意图保留） | Google Docs 级 |

---

## 3. 选型决策：乐观锁 CAS（Phase 3 实现目标）

### 3.1 设计

**Neo4j 节点新增属性**：`version: int64`（初始值 0）

**UpdateContent Cypher**：
```cypher
MATCH (n:Node {id: $id})
WHERE n.version = $expected_version OR $expected_version = -1
SET n.content = $content,
    n.updated_at = $updated_at,
    n.version = coalesce(n.version, 0) + 1
RETURN n.id, n.version
```
- `expected_version = -1` 表示强制覆盖（管理员/迁移用途）
- 若 WHERE 不匹配（0 rows）→ `domain.ErrConflict`

**API 变更**：

```
PATCH /api/v1/nodes/:id/content
Body: { "content": "...", "version": 3 }
Response 200: { "version": 4 }
Response 409: { "error": "conflict", "current_version": 5, "current_content": "..." }
```

**前端行为**：
1. GET 节点时缓存 `version`
2. 提交编辑时携带 `version`
3. 收到 409 → Toast 提示"内容已被他人修改，请刷新后重新编辑"
4. WS `node_updated` 事件携带新 `version`，前端更新本地缓存

### 3.2 改动清单

| 层 | 文件 | 改动 |
|----|------|------|
| domain | `domain/node.go` | `Node` 加 `Version int64` |
| store | `store/repo_node.go` | `cypherUpdateNodeContent` 加 version CAS；`scanNode` 读 version；`UpdateContent` 参数加 `expectedVersion` |
| store | `store/repo_node.go` | `GetByID` RETURN 加 `n.version` |
| service | `service/node.go` | `UpdateNodeContentInput` 加 `Version int64` |
| api | `api/http/handlers_node.go` | `updateContentReq` 加 `Version`；冲突时返回 409 |
| frontend | `src/api/client.ts` | `updateNodeContent` 加 `version` 参数 |
| frontend | `src/pages/Home.tsx` | 编辑提交携带 version；409 Toast 提示 |

### 3.3 向后兼容

- 旧节点无 `version` 属性 → `coalesce(n.version, 0)` 处理，首次更新自动写入 version=1
- 旧前端（不传 version）→ 服务端可拒绝（400）或兼容（视为 expected_version = any）

## 4. 后续演进：CRDT（M4）

当同时在线用户增加（>10人/湖），考虑引入 **Yjs** CRDT：
- 节点 content 改为 Yjs `Y.Text` 类型
- 后端存储 Yjs Doc binary state（附件表或单独的 `node_doc_states` 表，已在 migration 0010 预留）
- WS 推送 Yjs update binary，而非全量 content
- 参考：`backend-go/cmd/yjs-bridge/`（已有 YJS bridge 骨架）
