# G11 · 可靠性与容灾设计

**版本：** v1.0
**日期：** 2026‑04‑21
**适用对象：** SRE / 后端 / 架构
**核心隐喻：** 防汛抗旱 + 河道改道

> 没有 100% 可用的系统，只有不同等级的"湿身"概率。

## 二、故障域划分

| 域 | 隔离手段 | 爆炸半径 |
| :--- | :--- | :--- |
| 单实例故障 | 多副本 + LB 健康检查 | 0 用户感知 |
| 单 AZ 故障 | 跨 AZ 部署（≥ 2 AZ） | < 5min 切换 |
| 单 Region 故障 | 异地灾备（DR Region） | RPO ≤ 15min, RTO ≤ 1h |
| LLM 供应商故障 | 多 Provider + 三态降级 | 仅 AI 功能受影响 |
| 数据损坏 | 每日全量备份 + 7d/30d/1y 三档 | RPO ≤ 24h |

## 三、数据可靠性

### 3.1 PostgreSQL
- 主从同步复制（synchronous_commit=on）
- 流复制 + WAL 归档到 S3
- PITR 恢复窗口：30 天
- 每周演练全量恢复

### 3.2 Neo4j
- Causal Cluster（≥ 3 Core）
- 事务日志归档
- 每日 `neo4j-admin dump` → S3
- 测试集群定期演练 restore

### 3.3 对象存储
- S3 跨 Region 复制（CRR）
- Versioning + Lifecycle（详见 G4）
- 删除保护：MFA Delete

### 3.4 Redis（缓存 + Outbox）
- Sentinel 主备
- AOF + RDB 双持久化
- Outbox 在 PG 中是真源；Redis Stream 仅做加速通知

---

## 四、Outbox 与事务一致性

- 业务写入 PG = 写入 Outbox 在同一 transaction
- 后台 publisher 单独进程消费 Outbox → Redis Stream → AI 工作流
- **保证：** at-least-once；下游消费者必须**幂等**（按 `event_id` 去重）

---

## 五、降级链路汇总

| 故障 | 降级行为 | 用户文案 |
| :--- | :--- | :--- |
| LLM 主模型超时 | 切到副模型（standard tier） | （无感知） |
| 副模型也失败 | 切到 fallback：仅本地 Embedding 相似度 | "AI 暂歇" |
| 全部 AI 失败 | 关闭暗流推断；保留手动连线 | "潮汐异常" |
| Neo4j 不可用 | 进入"只读模式"：可看不可写 | "水位过低，暂停投石" |
| WebSocket 不可用 | 退化为 5s polling | "信号微弱，正在重连" |
| 文件上传失败 | 客户端缓存重试 3 次 → 提示用户 | "水流被堵，请重试" |

## 六、混沌工程（Chaos Engineering）

每月强制演练：

1. **Pod 随机杀死**：验证滚动恢复
2. **AZ 故障注入**：切流到备 AZ
3. **LLM 注入 500 错误**：验证降级链
4. **网络分区**：Neo4j 集群脑裂演练
5. **磁盘填满**：备份 / 日志清理策略验证

工具：Chaos Mesh / LitmusChaos。
