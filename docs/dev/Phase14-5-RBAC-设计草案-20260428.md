# Phase 14.5 平台管理员 RBAC 设计草案

日期：2026-04-28  
状态：14.5-A/B 已实现（数据表 / 仓库 / 集中鉴权 helper / OWNER 管理 API）

## 1. 背景

当前平台管理员通过 `RIPPLE_ADMIN_EMAILS` 配置白名单识别，适合 staging / MVP：

- 无需新增表即可快速交付管理员总览、灰度名单与灰度审计；
- 部署侧可通过环境变量快速授予 / 收回权限；
- 缺点是不可审计授权变更，且不适合企业版多管理员治理。

Phase 14.5 的目标是设计后续替换路径，不在本轮直接破坏兼容。

## 2. 设计目标

1. 保持 `RIPPLE_ADMIN_EMAILS` 兼容，迁移期间不影响 staging 管理员。
2. 平台管理员授权可审计、可撤销、可查询。
3. 鉴权逻辑集中，避免各 handler 重复判断邮箱。
4. 数据结构支持后续企业版：多角色、操作者、过期时间、备注。

## 3. 数据模型草案

```sql
CREATE TABLE platform_admins (
    user_id    UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    role       TEXT NOT NULL DEFAULT 'ADMIN' CHECK (role IN ('ADMIN', 'OWNER')),
    note       TEXT NOT NULL DEFAULT '',
    created_by UUID REFERENCES users(id) ON DELETE SET NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    revoked_at TIMESTAMPTZ
);

CREATE INDEX idx_platform_admins_active ON platform_admins (created_at DESC) WHERE revoked_at IS NULL;
```

## 4. 鉴权规则

平台管理员判定顺序：

1. 若用户 email 命中 `RIPPLE_ADMIN_EMAILS`，允许访问（兼容兜底）；
2. 否则查询 `platform_admins` 中 `user_id = actor.id AND revoked_at IS NULL`；
3. 平台管理员运营端点仅接受 JWT 用户会话，不允许 API Key 继承管理员权限；
4. 未来若需要 OWNER / ADMIN 分层：
   - `OWNER` 可授予 / 撤销平台管理员；
   - `ADMIN` 可使用运营台、灰度名单、审计查询，但不可管理平台管理员。

## 5. API 草案

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| `GET` | `/api/v1/admin/platform_admins` | OWNER | ✅ 列出平台管理员 |
| `POST` | `/api/v1/admin/platform_admins` | OWNER | ✅ 按 email / user_id 授权 |
| `DELETE` | `/api/v1/admin/platform_admins/{user_id}` | OWNER | ✅ 软撤销 |

所有写操作写入 `audit_logs`：

- `resource_type=platform_admin`
- `action=platform_admin.grant` / `platform_admin.revoke`
- `resource_id=<user_id>`

## 6. 迁移计划

1. ✅ Sprint 14.5-A：新增表、仓库与集中鉴权 helper，但继续保留 `RIPPLE_ADMIN_EMAILS`；
2. ✅ Sprint 14.5-B：后台 API 管理平台管理员，写审计；
3. Sprint 14.5-C：Settings 增加只读管理员列表；
4. 企业版前：将 `RIPPLE_ADMIN_EMAILS` 降级为 bootstrap owner 机制，仅用于首次初始化。

## 7. 风险与约束

- 不做硬切换，避免 staging 管理员失权。
- 不允许物理删除管理员授权记录；撤销必须保留审计链。
- 平台管理员管理 API 必须使用 OWNER 角色，不允许普通 ADMIN 自提权。
