# Phase 14.6 运营化回归与准入设计草案

日期：2026-04-28  
文档层级：`docs/dev/*.md`，承接 `docs/system-design/Phase14-规划草案.md` 与 Phase 14.5 RBAC / WebSocket 修复。

## 1. 背景

Phase 14 已完成 Org 配额、运营台、灰度名单、平台管理员 RBAC 与 staging WebSocket 修复。当前主要风险不再是单个功能缺失，而是发布前缺少稳定的“运营化回归准入”：

- Settings 管理子 Tab 与 RBAC 面板需要可重复 E2E；
- LakeWS / yjs-bridge 曾因 nginx 反代与 OriginPatterns 配置出现 staging-only 故障；
- staging smoke 仍依赖手工临时用户、手工 token 与浏览器操作；
- SOP 已记录，但尚未形成一键化准入脚本与 CI / 手工门禁清单。

## 2. 目标

Phase 14.6 聚焦“运营化回归与准入 Gate”，不新增业务大功能：

1. 固化运营台关键路径 E2E：Settings 子 Tab、平台管理员 RBAC、灰度名单、审计入口；
2. 固化 WebSocket staging smoke：`/yjs` 反代、LakeWS Origin、ws-only token；
3. 形成 staging 临时数据清理脚本，避免 smoke 污染环境；
4. 更新发布准入清单，让 Phase 14 从“可用”进入“可持续发布”。

## 3. 非目标

- 不在 14.6 引入 GraphQL；
- 不在 14.6 扩展计费支付闭环；
- 不改平台管理员对外 API 语义；
- 不把 staging 专用凭据写入仓库。

## 5. 接口与脚本设计

### 5.1 LakeWS 自动化回归

本地单测层：

- `websocketOriginPatterns()` 将完整 CORS URL 转换为 host pattern；
- 使用 `httptest.Server` + `nhooyr.io/websocket.Dial` 验证 `Origin: http://fn.cky:14173` 可完成握手；
- yjs-bridge 使用同一转换规则，避免主 WS 与协作 WS 行为分叉。

Staging smoke 层：

- `/yjs` 普通 HTTP 请求期望 `400 lake required`，证明没有落到 SPA fallback；
- LakeWS 对有权限湖使用 Upgrade 请求期望 `101 Switching Protocols`；
- yjs-bridge 携带 ws-only token 期望 `101 Switching Protocols`。

### 5.2 Settings E2E

Playwright 通过 API mock 建立登录态，不依赖真实后端：

- 登录页 mocked `/api/v1/auth/login`；
- 首页 mocked `lakes/spaces/recommendations`；
- Settings mocked `admin/overview`、`admin/platform_admins`、`api_keys`、`admin/graylist`、`audit_logs`；
- 验证 5 个子 Tab 均能切换到对应标题。

### 5.3 Staging 数据清理

清理范围：

- 临时 graylist / smoke 用户：按 smoke email 前缀删除；
- 临时 Neo4j Lake 投影：按 `ws-smoke-` / `ws-curl-` / `ws-origin-` 名称前缀删除；
- 不删除真实用户或真实运营数据；
- 清理脚本只在 staging 执行，生产环境必须改为只读审计预览 + 人工确认。

## 6. 验收标准

- `go test -race -count=1 ./...` 通过；
- `go vet ./...` 通过；
- `npm.cmd run lint` 通过；
- `npm.cmd test -- --run tests/platformAdminRegression.test.ts` 通过；
- `npm.cmd run build` 通过；
- `npm.cmd run e2e -- e2e/settings-tabs.spec.ts --reporter=list` 在 Vite dev server 下通过；
- staging：`/yjs` 返回 yjs-bridge 错误而非 SPA HTML；
- staging：LakeWS 与 yjs-bridge Upgrade smoke 返回 `101`；
- staging：临时 smoke 数据清理完成。

## 7. 风险与防护

| 风险 | 防护 |
|------|------|
| Playwright 依赖本地 dev server | E2E 运行前显式启动 Vite 5234，结束后关闭 |
| staging 凭据泄露 | 凭据只保存在 session memory / 运行时，不写入仓库 |
| 清理脚本误删数据 | 只按固定 smoke 前缀删除；生产禁用自动删除 |
| WS Origin 规则再次误配 | 单测覆盖 `nhooyr` host pattern 语义；SOP 写明完整 URL 与 host pattern 差异 |

## 8. 推荐执行顺序

1. 先合入 LakeWS / yjs Origin 单测与 Settings E2E；
2. 再补 staging smoke 数据清理脚本；
3. 更新 Phase 14 规划与发布准入清单；
4. 若 CI 稳定，再评估把 Playwright E2E 纳入可选 CI job。
