# Phase 14.2 配额拦截点接入记录（2026-04-28）

> 文档层级：`docs/dev/*.md`，承接 `docs/system-design/Phase14-规划草案.md` 与 `docs/dev/Phase14-A-技术方案评审-20260427.md`。

## 结论

Sprint 14.2 已将组织配额从“可读 / 可调 / 可审计”推进到核心写路径拦截。超限统一返回领域错误 `ErrQuotaExceeded`，HTTP 层映射为 `429 Too Many Requests`，避免静默失败。

## 接入范围

| 配额项 | 拦截点 | 说明 |
|--------|--------|------|
| `max_members` | `OrgService.AddMember` | 邀请 / 添加成员前按当前组织成员数校验。|
| `max_lakes` | `LakeService.SetLakeOrg` | 湖绑定到组织时校验该组织已有湖数量；无组织归属的普通建湖不计入组织配额。|
| `max_nodes` | `NodeService.Create` / `BatchImportNodes` | 写入前按湖归属组织统计已有节点数；组织湖内单节点与批量导入均受控。|
| `max_attachments` | `AttachmentHandlers.Upload` | 上传绑定节点的附件时解析节点所属湖与组织并校验附件数量。|
| `max_storage_mb` | `AttachmentHandlers.Upload` | 上传绑定节点的附件时按组织附件总字节数向上取整为 MB 校验。|
| `max_api_keys` | `APIKeyHandlers.Create` | 请求提供 `org_id` 时校验组织成员身份与 API Key 数量。|

## 数据模型补充

新增迁移：

- `backend-go/migrations/0018_resource_org_scope.up.sql`
- `backend-go/migrations/0018_resource_org_scope.down.sql`

变更：

- `api_keys.org_id`：支持组织级 API Key 配额统计；
- `attachments.org_id`：支持组织级附件数量与容量统计；
- 为两列增加非空组织索引，避免运营查询全表扫描。
- Neo4j 增加 `Lake.org_id` 索引，组织节点计数改为基于 `Node.lake_id` 与 `Lake.org_id` 匹配，避免陈旧 `CONTAINS` 关系放大计数。

## 前端运营台

`OrgPanel` 新增 `Quota` Tab：

- 成员可读组织配额；
- OWNER / ADMIN 可编辑 `max_members`、`max_lakes`、`max_nodes`、`max_attachments`、`max_api_keys`、`max_storage_mb`；
- 使用现有 `GET/PATCH /api/v1/organizations/{id}/quota` 契约；
- 保存失败时直接展示后端错误，避免静默失败。

`APIKeyManager` 新增组织归属选择：

- 创建 API Key 时可选择个人 Key 或组织 Key；
- 组织 Key 会提交 `org_id` 并受 `max_api_keys` 约束；
- 列表展示组织归属短 ID，便于运营排查。

## 审查要点

1. API Key 与附件补充 `org_id`，避免只能按用户维度统计导致组织配额失真；
2. 节点配额在 service 层统一拦截，覆盖单节点创建和批量导入；
3. `Condense` 覆盖 MIST 凝露路径，并避免同湖 / 同组织节点重复计入新增配额；
4. 附件容量按总量 MB 向上取整，避免单文件分别取整导致过度拒绝；
5. 附件跨 scope 同 SHA 不再静默复用旧元数据，避免组织附件绕过配额；
6. Lake 配额放在 `SetLakeOrg`，与当前“建湖后绑定组织”的既有 API 保持兼容；
7. 旧数据 `org_id=''` 保持兼容，不强制迁移历史个人资源。
