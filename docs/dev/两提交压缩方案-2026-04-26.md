# 两提交压缩方案（docs + code）

目的：将当前 4 提交压缩为 2 提交，便于审阅。

当前提交顺序（从旧到新）：

1. `2e49453` docs
2. `3c4addd` chore
3. `2da0a15` fix
4. `83d7dae` feat

---

## 安全前置（必须）

先打保护分支，避免误操作：

```bash
git checkout main
git branch backup/pre-squash-2026-04-26
```

---

## 目标结构

- 提交 A（docs）：保留 `2e49453`
- 提交 B（code）：把 `3c4addd + 2da0a15 + 83d7dae` 压成一个

推荐新提交信息：

- `docs: 文档重命名收敛与引用修复`
- `feat: 扩展组织通知模板与分享能力并包含修复项`

---

## 执行步骤（本地）

```bash
# 1) 回到 docs 提交点，并保留后续改动到暂存区
git reset --soft 2e49453

# 2) 重新提交 code 聚合提交
git commit -m "feat: 扩展组织通知模板与分享能力并包含修复项"
```

执行后历史将变为：

- `2e49453` docs...
- `<new_code_commit>` feat...

---

## 校验步骤

```bash
git log --oneline -5
git show --stat -1
git status
```

期望：

- 仅保留 2 个业务提交（docs + code）。
- `git status` 为空。

---

## 远端同步策略

如果该分支已经推送过，压缩后需要强推：

```bash
git push --force-with-lease origin main
```

说明：`--force-with-lease` 比 `--force` 更安全，会校验远端是否有他人新提交。

---

## 快速回滚

若压缩过程不符合预期，可直接回到保护分支：

```bash
git reset --hard backup/pre-squash-2026-04-26
```

或使用 reflog 回退。

---

## 注意

- 如果 `main` 是受保护分支，建议在新分支上执行压缩并发起 PR，而不是直接改写 `main` 历史。
- 如果团队已有基于旧提交的开发分支，请先同步沟通再 force push。
