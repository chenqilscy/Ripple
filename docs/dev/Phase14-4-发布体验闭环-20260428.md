# Phase 14.4 发布与体验闭环

日期：2026-04-28  
状态：启动

## 1. 目标

Phase 14.4 聚焦把 P14-A「Org 配额与运营台」从功能完成推进到可发布、可运营、可回归：

1. 体验方试用：覆盖普通用户、组织管理员、平台管理员三类入口。
2. 配额文档：说明默认免费配额、触发点、错误语义与运营调整方式。
3. 运营 SOP：灰度名单、管理员总览、组织配额调整、审计追踪与回滚。
4. 准入清单：后端、前端、staging、CI、数据安全与性能观察项全部过线。

## 2. 体验方试用脚本

### 2.1 普通用户

- 注册 / 登录。
- 创建湖、创建节点、批量导入节点。
- 被组织配额拦截时，应看到明确错误而不是静默失败。

### 2.2 组织管理员

- 打开组织面板。
- 在组织列表查看 `Members/Lakes/Nodes` usage 摘要与 `Latest audit`。
- 进入 `Quota` Tab，查看 usage 进度条与 `Recent quota audits`。
- 展开完整审计日志，确认默认查询 `resource_type=org_quota`。

### 2.3 平台管理员

- 打开 `Settings`。
- 管理员总览展示组织数、用户数、灰度邮箱数与组织 quota usage。
- 灰度名单完成新增、更新、删除闭环。
- 非平台管理员只能看到权限提示，不能编辑灰度名单。

## 4. 运营 SOP

### 4.1 灰度名单

1. 后端通过 `RIPPLE_REGISTRATION_GRAYLIST_ENABLED=true` 开启注册灰度。
2. 平台管理员邮箱通过 `RIPPLE_ADMIN_EMAILS` 配置。
3. 平台管理员在 `Settings` → `灰度名单` 添加允许注册邮箱。
4. 新增 / 更新同一邮箱时以邮箱唯一键 upsert，备注以后一次提交为准。
5. 移除邮箱前应确认该用户是否仍处于邀请窗口。

### 4.2 配额调整

1. 组织管理员进入组织 `Quota` Tab。
2. 调整 `Members/Lakes/Nodes/Attachments/API Keys/Storage` 限额。
3. 保存后刷新 overview，确认 usage 与 `Recent quota audits` 同步更新。
4. 若误操作，依据审计日志中的变更前后 detail 手工恢复。

### 4.3 管理员总览

1. 管理员总览以 `GET /api/v1/admin/overview` 聚合平台运营态。
2. 当前限制为最多返回 50 个组织，适合作为 MVP 运营台。
3. 若组织数接近 50 或响应变慢，应启动批量聚合 / `org_usage_snapshots` 优化。

## 6. 代码审查结论

### 6.1 已修复

- 前端 `recent_quota_audits` / `organizations` 缺省数组访问导致的 runtime crash。
- 灰度名单排序从字符串比较改为时间戳比较。
- 文档中移除具体 staging 测试账号，改为泛称平台管理员测试账号。
- 新增 Vitest 回归测试，覆盖缺省 `organizations` 与缺省 `recent_quota_audits`。
- 补齐前端 ESLint 依赖与 flat config，恢复 `npm run lint` 可执行性。

### 6.2 进入 14.4 观察项

- 管理员总览当前按组织循环聚合 usage，MVP 可接受；组织数继续增长时改为批量聚合或使用 `org_usage_snapshots`。
- 平台管理员权限当前采用 `RIPPLE_ADMIN_EMAILS` 配置白名单，适合 staging / MVP；正式企业版前评估独立平台管理员角色表。
- `created_by` 外键采用保留历史优先策略，当前不做级联删除；若未来支持用户物理删除，需要补匿名化 / SET NULL 迁移方案。
- 前端 `lint` 已作为发布 gate 恢复；Fast Refresh 辅助导出与历史 effect 依赖模式已在配置中降噪，后续若要收紧规则需先分批改造存量组件。

## 7. 回滚方案

1. 前端问题：回滚 nginx 静态资源到上一个 `dist` 包或重新部署上一 commit 构建产物。
2. 后端问题：使用 staging compose 回滚到上一镜像 / commit，并停止新注册灰度开关。
3. 数据问题：`0019_graylist.down.sql` 可删除灰度名单表；执行前必须导出灰度邮箱清单，避免丢失运营数据。
