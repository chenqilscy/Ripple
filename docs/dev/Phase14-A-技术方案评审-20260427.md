# Phase 14-A Org 配额数据模型技术方案评审（2026-04-27）

## 结论

Phase 14-A 第一切片采用“PG 强一致配额表 + 服务层权限校验 + 审计日志”的方案，先交付组织配额数据模型、仓储、服务与 HTTP 管理端点，业务拦截点在后续 Sprint 14.2 接入。

## 十角色评审摘要

【需求创造师】: 组织需要从协作容器升级为可运营对象，配额是灰度、商业化和安全边界的共同基础。

【产品经理 (PM)】: 优先做 Org 配额与运营台，不引入真实支付；第一切片只要求可读、可调、可审计。

【接口设计师】: 新增 `GET /api/v1/organizations/{id}/quota` 与 `PATCH /api/v1/organizations/{id}/quota`；读要求组织成员，写要求 ADMIN+。

【正方（架构师）】: 配额持久化在 PostgreSQL，使用 `org_quotas` 单行保存默认免费层上限；保留 `org_usage_snapshots` 作为后续运营报表基础。

【反方（审查员）】: migration 必须使用当前最新序号之后的 `0017`，不能复用历史 `0014`；`org_quota` 审计查询必须映射回组织权限；配额检查要防止整数溢出。

【决策者（项目经理）】: 裁决采用该方案，本轮实现第一切片，不一次性接入所有业务拦截点，避免范围失控。

## 数据模型

新增迁移：

- `backend-go/migrations/0017_org_quotas.up.sql`
- `backend-go/migrations/0017_org_quotas.down.sql`

新增表：

1. `org_quotas`
   - `org_id`
   - `max_members`
   - `max_lakes`
   - `max_nodes`
   - `max_attachments`
   - `max_api_keys`
   - `max_storage_mb`
   - `created_at`
   - `updated_at`
2. `org_usage_snapshots`
   - 保留给后续运营台和趋势报表使用。

## 服务边界

新增领域对象：

- `domain.OrgQuota`
- `domain.OrgUsageSnapshot`
- `domain.OrgQuotaKey`
- `domain.ErrQuotaExceeded`

新增服务能力：

- `OrgService.GetQuota()`：成员可读；
- `OrgService.UpdateQuota()`：ADMIN+ 可写，并写入审计日志；
- `OrgService.CheckQuota()`：后续业务拦截点统一调用，超限返回 `ErrQuotaExceeded`。

## HTTP 契约

```text
GET   /api/v1/organizations/{id}/quota
PATCH /api/v1/organizations/{id}/quota
```

`PATCH` 请求体字段均可选，但至少提供一个字段：

```json
{
  "max_members": 30,
  "max_lakes": 50,
  "max_nodes": 10000,
  "max_attachments": 1000,
  "max_api_keys": 10,
  "max_storage_mb": 1024
}
```

## 审查修复

1. 逻辑正确性：补齐 `org_quota` 审计查询权限映射；
2. 边界条件：`CheckQuota` 增加 `used + delta` 溢出保护；
3. 架构一致性：使用 `0017_org_quotas`，保持 migration 顺序；
4. 安全性：配额读要求成员，写要求 ADMIN+；
5. 性能：`EnsureDefault` 改为先读，不存在才插入默认行，避免每次读都走写路径。
