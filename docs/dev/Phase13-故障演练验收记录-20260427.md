## yjs-bridge 修复说明

第一次 yjs-bridge 演练暴露的是恢复链路缺陷，而不是业务逻辑缺陷：

1. compose 使用 `node:20-alpine`，并在启动时执行 `apk add --no-cache go && go run ./cmd/yjs-bridge`；
2. 远端 Alpine 拉包出现 TLS 错误，导致容器反复 `Restarting`；
3. 后续排查又发现 staging compose 与 Go 程序存在两处环境变量错配：
   - JWT 配置写成了 `YJS_BRIDGE_JWT_SECRET`，而程序读取的是 `RIPPLE_JWT_SECRET`；
   - Redis 配置写成了 `YJS_BRIDGE_REDIS_ADDR/PASS`，而程序读取的是 `YJS_BRIDGE_REDIS_URL`。

已完成修复：

1. `yjs-bridge` 改为 `golang:1.23-alpine` 镜像，启动命令改为 `/usr/local/go/bin/go run ./cmd/yjs-bridge`；
2. JWT 配置统一为 `RIPPLE_JWT_SECRET`；
3. Redis 配置统一为 `YJS_BRIDGE_REDIS_URL=redis://:${REDIS_PASSWORD}@redis:6379/0`。

修复后验证结果：

- `http://fn.cky:17790/healthz` 返回 `200 OK`，响应体为 `ok`；
- 容器镜像为 `golang:1.23-alpine`，状态为 `Up`；
- 日志出现 `yjs-bridge: redis pub/sub enabled` 与 `yjs-bridge listening on :7790`；
- 之后再次执行 Phase 13 smoke，结果全绿。
