## 1. 候选主题清单

### 候选 A · 性能与可观测性深化

**目标**：将 Ripple 从「能跑」推到「能用得很快、问题能查出来」。

**关键产出**：
- WebSocket Hub 压测基线（10k 连接 / 1k QPS msg）
- DB 慢查询审计 + index 建议（pg_stat_statements）
- 全链路 trace（OpenTelemetry → Jaeger）
- 前端 Web Vitals 监控（LCP / INP / CLS）
- pprof 自动化截图归档（每日一次）

**预估工作量**：3-4 周。
**收益**：解决 Phase 12-14 持续暴露的「症状不明」问题；为商业化扩量打基础。
**风险**：技术债不少，可能拖到 5-6 周。

---

### 候选 B · 多端协作 / 移动端

**目标**：把"造浪池"做出真正的多端协作体验，移动端读优先。

**关键产出**：
- React Native 或 Capacitor 原型（iOS/Android）
- 离线编辑 + 冲突合并（基于 Yjs Y.Doc 持久化）
- 推送通知（FCM/APNs 接入 notifications 表）
- 手机端造浪池触摸交互优化

**预估工作量**：6-8 周（含 RN 学习曲线）。
**收益**：扩大用户基数；移动端是知识工作者刚需。
**风险**：RN 与现有 Three.js / WebGL 渲染栈兼容性未验证；可能要做大重构。

---

### 候选 C · AI Workflow 扩展

**目标**：把 LLM 能力从单点（已 6 家 provider）扩到工作流编排。

**关键产出**：
- 节点类型扩展：AI Agent 节点（能跨节点取上下文）
- Prompt 模板库（用户可保存/分享）
- AI 自动整理：选中多个节点 → AI 自动归类 / 生成摘要 / 生成连线
- LLM 路由策略可视化（基于 LLMRouter 已有架构）

**预估工作量**：4-5 周。
**收益**：直接对应"AI 应用专家"角色诉求；商业化有明确付费点（按调用计费）。
**风险**：UX 设计是关键，做不好就是技术 demo。

---

### 候选 D · 商业化与计量

**目标**：上线收费能力，把 Ripple 从工具变产品。

**关键产出**：
- 订阅计费集成（Stripe / 国内：微信/支付宝）
- org_quotas 真实计量（已有表，未接入扣减）
- LLM 调用计费（llm_calls 表 → 月度账单）
- 团队工作区（spaces 已有，需补 billing 维度）
- 试用转付费转化漏斗埋点

**预估工作量**：5-6 周（含支付合规）。
**收益**：开始有营收；验证 PMF。
**风险**：合规复杂（特别是国内）；可能需要外部法务支持。

---

### 候选 E · 安全与合规加固

**目标**：满足 SOC2 / 等保 2.0 等准入门槛。

**关键产出**：
- 端到端审计（audit_events 已有，需补 export 与 SIEM 接入）
- 静态代码安全扫描（gosec / semgrep）
- 依赖漏洞扫描（govulncheck / npm audit）固化到准入流
- Secret 管理（Vault / 自建）
- 数据加密（at-rest + in-transit）核查

**预估工作量**：3-4 周。
**收益**：To B 客户必备；不做就拿不到大客户。
**风险**：纯成本投入，短期无新功能。

## 3. 推荐路径

**主线**：候选 C（AI Workflow）+ 候选 D（商业化）打包做 Phase 15
- AI Workflow 是产品差异化卖点
- 商业化是验证 PMF 必经之路
- C + D 形成「能力 → 收钱」闭环

**辅线**（Phase 15.5）：候选 A（性能）零散插入，每个迭代留 20% 时间做技术债

**延后**：候选 B（移动端）等 PMF 验证后再投入；候选 E（合规）等首个 To B 大客户出现时启动
