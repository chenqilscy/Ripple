# 图谱价值增强架构设计

**所属项目：** Ripple (青萍)
**类型：** 架构设计文档
**版本：** v1.0
**日期：** 2026-04-30
**状态：** 待审查

---

## 一、问题背景

当前 LakeGraph.tsx 的定位是"知识图谱可视化工具"——用户将节点扔进去，图谱展示节点分布。功能上正确，但价值感不足：

- 用户"看"到结构，但感受不到价值
- 无法回答"我的知识有什么缺口"或"这个概念和那个有什么关系"
- 工具变成了展示品，而不是思考和发现的助手

**核心问题：图谱是可视化工具，而不是探索工具。**

---

## 二、设计目标

| 维度 | 现状 | 目标 |
|------|------|------|
| **定位** | 知识图谱可视化 | 探索工具 + 规划工具 |
| **价值感** | 用户看节点分布 | 用户发现惊喜 + 规划下一步 |
| **交互核心** | 节点展示 + 缩放拖拽 | 智能推荐驱动探索 |

**两个核心价值锚点：**

1. **发现（Discovery）**：帮用户找到自己还没意识到的关联——从"我有什么"到"我原来不知道这些有关系"
2. **规划（Planning）**：帮用户决定接下来学什么/做什么——从"我积累了什么"到"我接下来应该做什么"

**三个必备支撑功能：**

- **路径追溯**：展示两个节点之间关联链路的透明推理
- **智能推荐**：AI 自动发现潜在但尚未建立的关联
- **聚类视图**：将相似节点自动聚合，帮助用户从宏观上理解知识结构

---

## 三、使用场景分析

用户进入图谱有三种典型意图：

| 意图 | 描述 | UI 响应 |
|------|------|--------|
| **录入** | 进来快速记录几个想法，不看图谱 | 图谱成为"笔记本"——快速添加，自动归类 |
| **浏览** | 漫无目的地翻翻，没有明确目标 | 被动获得惊喜——节点高亮提示潜在关联 |
| **探索** | 带着目标来找关联、做规划 | 系统引导发现——推荐路径和下一步建议 |

三种意图并存，UI 需要同时服务所有场景。

---

## 四、整体架构

### 4.1 架构理念

将"发现"和"规划"能力直接嵌入图谱交互，使图谱从可视化工具升级为探索工具。

### 4.2 模块划分

```
┌──────────────────────────────────────────────────────┐
│                    前端（LakeGraph.tsx）               │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │   图谱视图    │  │   发现引擎    │  │   规划器    │ │
│  │  （已有，改造）│  │    （新增）    │  │   （新增）   │ │
│  └──────────────┘  └──────────────┘  └────────────┘ │
└──────────────────────────────────────────────────────┘
          WebSocket / SSE 实时推送
┌──────────────────────────────────────────────────────┐
│                    后端分析服务                         │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │  实时推荐引擎  │  │  路径分析引擎  │  │  缺口分析引擎│ │
│  └──────────────┘  └──────────────┘  └────────────┘ │
│  ┌──────────────┐  ┌──────────────┐                  │
│  │  聚类分析引擎  │  │  规划推荐引擎  │                  │
│  └──────────────┘  └──────────────┘                  │
└──────────────────────────────────────────────────────┘
```

### 4.3 降级策略

| 实时分析状态 | 降级方案 |
|------------|---------|
| 实时分析运行正常 | 实时推送推荐结果 |
| 实时分析失败 | 降级到按需触发（用户主动请求） |
| 批处理失败 | 降级到静态推荐（基于已有关联历史） |

### 4.4 计算位置

**纯后端计算，前端只负责展示。** 后端负责分析引擎和推送，前端作为纯粹的展示层。实时性依赖 WebSocket/SSE 推送能力。

## 五、交互模型

### 5.1 混合交互模式

采用"被动轻量提示 + 主动完整功能"的混合模式：

- **被动态（浏览）**：节点显示小徽章或微高亮，提示存在潜在关联
- **主动态（探索）**：用户悬停/点击节点，展开完整推荐面板

### 5.2 发现模式交互流程

```
用户浏览图谱
    ↓
后端实时分析节点关联（后台持续运行）
    ↓
节点上显示轻量提示（徽章数量 = pending 推荐数）
    ↓
用户悬停/点击节点
    ↓
展开"发现面板"：
  - 发现 N 条潜在关联（附推荐理由）
  - 点击"查看路径" → 路径可视化
  - 点击"忽略" → 隐藏该推荐
    ↓
用户确认关联 → 建立正式连线
```

**关键原则：**
- 推荐理由透明可见（不只是"推荐"，还要说明"为什么"）
- 用户始终掌控主导权（可以忽略、调整、接受）
- 发现是"意外惊喜"，不是"系统打扰"

### 5.3 规划模式交互流程

```
用户点击"规划"入口
    ↓
系统分析：
  1. 已有关联的结构完整性
  2. 孤岛节点（没有连线）
  3. 语义空白（某领域节点稀疏）
    ↓
展示"规划面板"：
  - 知识缺口地图（哪些领域是空白）
  - 下一步建议（推荐添加什么节点）
  - 探索路径（哪些节点可以帮助理解某主题）
    ↓
用户选择建议 → 一键创建节点
```

**关键原则：**
- 规划基于已有数据（不凭空推荐）
- 建议具体化（如"建议添加：分布式系统基础概念"）
- 操作闭环（从建议到创建节点，一键完成）

### 5.4 交互状态流转

```
[浏览态] ← 悬停节点 → [发现悬停] ← 点击节点 → [发现展开]
   ↑                                  ↓
   ← 退出发现 ← 点击画布空白 / ESC ← [路径追踪]
   ↑
[规划态] ← 点击规划入口 ← 从任何状态

节点录入：任何状态下，按 N 或点击 "+" 按钮 → 快速创建
```

## 六、核心功能模块

### 6.1 模块一：智能推荐引擎（发现核心）

**推荐触发时机：**
- 节点新增时 → 自动分析该节点的潜在关联
- 节点内容更新时 → 重新分析相关节点
- 定时全量扫描 → 发现被忽略的潜在关联

**推荐数据结构：**
```typescript
interface Recommendation {
  id: string
  sourceNodeId: string
  targetNodeId: string
  reason: string           // 推荐理由："两者都涉及系统可用性"
  confidence: number       // 置信度 0-1
  createdAt: number
  status: 'pending' | 'accepted' | 'rejected' | 'ignored'
}
```

**前端展示：**
- 节点旁显示小圆点徽章（数量 = pending 推荐数）
- 悬停节点 → 出现推荐气泡（显示 1-2 条）
- 点击气泡 → 展开完整推荐面板

### 6.2 模块二：路径追溯（透明推理）

**路径计算逻辑：**
- 给定两个节点，找到它们之间的最短关联路径
- 返回路径上的所有中间节点和边
- 每条边的关联理由一并返回

**路径数据结构：**
```typescript
interface PathResult {
  sourceId: string
  targetId: string
  nodes: Array<{
    id: string
    title: string
    reason: string  // 该节点在路径中的作用
  }>
  edges: Array<{
    from: string
    to: string
    similarity: number
    kind: string
  }>
  totalSteps: number
}
```

**前端展示：**
- 路径上的节点和边高亮为"发现色"（青碧色渐变 #2E8B90）
- 路径经过的每条边标注简短理由
- 侧边面板展示路径叙事："A → [都涉及 X] → B"

### 6.3 模块三：聚类视图（结构洞察）

**聚类逻辑：**
- 基于节点内容和相似度进行聚类
- 每个聚类代表一个"知识领域"
- 识别聚类间的桥接节点（跨领域连接点）

**聚类数据结构：**
```typescript
interface Cluster {
  id: string
  label: string           // 领域名称（AI 生成）
  nodeIds: string[]
  color: string
  bridgeNodeIds: string[]  // 跨领域桥接节点
  density: number         // 领域饱满程度
}
```

**前端展示：**
- 聚类区域用不同颜色背景区分
- 桥接节点用特殊图标标注
- 点击聚类标签 → 聚焦该领域，隐藏其他聚类

### 6.4 模块四：规划模式（缺口分析）

**分析维度：**
1. **孤岛检测**：超过 N 天未更新且无关联的节点
2. **空白领域**：某语义领域完全没有节点
3. **断链风险**：仅有单向关联的节点对

**规划建议数据结构：**
```typescript
interface PlanningSuggestion {
  id: string
  type: 'add_node' | 'connect' | 'explore'
  title: string
  description: string
  priority: 'high' | 'medium' | 'low'
  relatedNodeIds: string[]
}
```

**前端展示：**
- 规划面板以卡片形式展示建议
- 优先级用颜色区分（高=红色，中=橙色，低=灰色）
- 一键接受 → 创建节点或建立关联

## 八、后端接口设计

### 8.1 WebSocket 事件（实时推送）

```typescript
// 推荐事件
{ type: 'recommendation', data: Recommendation }

// 聚类更新
{ type: 'cluster_update', data: Cluster[] }

// 规划建议
{ type: 'planning_suggestion', data: PlanningSuggestion }
```

### 8.2 REST API（按需请求）

| 方法 | 路径 | 描述 |
|------|------|------|
| POST | `/api/graph/path` | 计算两点间路径 |
| GET | `/api/graph/clusters` | 获取聚类结果 |
| GET | `/api/graph/planning` | 获取规划建议 |
| POST | `/api/graph/recommendation/:id/accept` | 接受推荐 |
| POST | `/api/graph/recommendation/:id/reject` | 拒绝推荐 |

## 九、实施路线图

### Phase 1：发现基础（2-4 周）

目标：让图谱开始有"发现"的价值

- 1.1 后端推荐引擎（实时分析基础版）
- 1.2 前端推荐徽章（节点旁显示数量）
- 1.3 推荐面板（展开查看推荐详情）
- 1.4 路径追溯（两点间路径计算 + 可视化）
- 1.5 降级机制（按需触发 + 批处理）

### Phase 2：聚类 + 规划（4-6 周）

目标：让图谱支持"结构洞察"和"规划"

- 2.1 聚类分析引擎
- 2.2 聚类视图（颜色区分 + 聚焦）
- 2.3 规划引擎（缺口分析）
- 2.4 规划面板（建议卡片）
- 2.5 一键接受建议（创建节点）

### Phase 3：协作增强（6-8 周）

目标：支持团队场景

- 3.1 推荐跨用户可见（团队知识共享）
- 3.2 规划面板团队版
- 3.3 发现历史（记录已发现/忽略的关联）
- 3.4 导出为报告（PNG + 路径说明）

## 十、风险与应对

| 风险 | 应对 |
|------|------|
| 推荐质量不高（误报多） | 用户可忽略/反馈，积累数据优化模型 |
| 实时分析性能问题 | Phase 1 先做按需触发，实时作为增强 |
| 后端改动大 | 推荐引擎独立部署，不影响现有功能 |
| 用户不信任推荐 | 提供透明的推荐理由，不做强制推荐 |

---

## 十一、设计决策总结

| 维度 | 决策 |
|------|------|
| **核心定位** | 图谱即探索工具（发现 + 规划） |
| **交互模式** | 混合——被动轻量提示 + 主动完整功能 |
| **分析架构** | 后端实时为主，降级按需/批处理 |
| **计算位置** | 纯后端，前端仅展示 |
| **实施路径** | Phase1 发现基础 → Phase2 聚类+规划 → Phase3 协作增强 |
| **与现有系统** | 增量叠加，不破坏现有功能 |

---

**关联问题：** P2-03（关联边缺少可视化）为本设计的已有基础设施，P2-01（图谱渲染性能风险）是本设计扩展到大规模节点的前置条件。
