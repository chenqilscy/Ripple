## 一、问题背景

当前 `PlanningPanel` 的"一键采纳"按钮调用 `AcceptPlanning` API，但后端是 TODO 空实现——按钮点了没反应。用户无法通过规划建议一键创建关联，Phase 2 的规划价值闭环缺失关键一环。

---

## 二、设计决策

| 决策点 | 选项 | 选择 |
|--------|------|------|
| **执行方式** | 全自动 vs 补充输入 | **全自动**：点击即执行，系统自动决定所有参数 |
| **add_node 内容来源** | 用户输入 vs LLM 自动生成 | **LLM 自动生成**：基于 description + 湖内现有节点内容生成 title/content/tags |
| **connect 边类型** | 用户选择 vs 固定值 | **固定 `relates`**：方向和边类型由系统自动决定，无需用户选择 |

---

## 三、整体架构

```
用户点击 "一键采纳"
    ↓
前端 RecommendationEngine.acceptPlanningSuggestion(fullSuggestion)
    ↓ POST /api/graph/planning/accept (body=完整 PlanningSuggestion)
后端 AcceptPlanning handler
    ↓ type 路由分发
  add_node → LLM 生成节点内容 → 创建节点 → 返回 nodeId
  connect  → 根据 related_node_ids 找目标节点 → 创建边 → 返回 edgeId
  explore  → 前端本地处理，后端仅返回确认
    ↓
前端收到成功响应 → Toast 提示 → 刷新图谱/面板
```

## 四、后端实现

### 4.1 路由

```
PUT /api/graph/planning/accept   ← 将 "accept" 从 GET 改为 PUT/POST
Content-Type: application/json
Body: 完整 PlanningSuggestion JSON（不是仅 id）
```

### 4.2 AcceptPlanning Handler 逻辑

```go
func (h *GraphAnalysisHandlers) AcceptPlanning(w http.ResponseWriter, r *http.Request) {
    var suggestion domain.PlanningSuggestion
    if err := json.NewDecoder(r.Body).Decode(&suggestion); err != nil {
        writeError(w, http.StatusBadRequest, "invalid request body")
        return
    }

    switch suggestion.Type {
    case "add_node":
        // 1. 获取湖内节点内容作为上下文
        lakeNodes, _ := h.getLakeNodeSummaries(suggestion.LakeId)
        // 2. LLM 生成节点内容
        title, content, tags := h.llm.GenerateNodeContent(
            suggestion.Description,
            lakeNodes,
        )
        // 3. 创建节点
        node, err := h.nodeSvc.CreateNode(&domain.CreateNodeRequest{
            LakeId: suggestion.LakeId,
            Title:  title,
            Content: content,
            Tags:   tags,
        })
        writeJSON(w, http.StatusOK, map[string]any{
            "id": suggestion.ID, "type": "add_node",
            "status": "accepted", "nodeId": node.ID,
        })

    case "connect":
        // 1. 确定源节点（relatedNodeIds[0]）
        // 2. 确定目标节点（relatedNodeIds[1] 或根据 title 关键词搜索）
        sourceId := suggestion.RelatedNodeIds[0]
        var targetId string
        if len(suggestion.RelatedNodeIds) > 1 {
            targetId = suggestion.RelatedNodeIds[1]
        } else {
            target, _ := h.findTargetNode(suggestion, sourceId)
            targetId = target.ID
        }
        if targetId == "" {
            writeError(w, http.StatusBadRequest, "target node not found")
            return
        }
        // 3. 检查边是否已存在（幂等）
        existing, _ := h.edgeSvc.GetEdge(sourceId, targetId)
        if existing != nil {
            writeJSON(w, http.StatusOK, map[string]any{
                "id": suggestion.ID, "type": "connect",
                "status": "accepted", "edgeId": existing.ID,
            })
            return
        }
        // 4. 创建边
        edge, err := h.edgeSvc.CreateEdge(&domain.CreateEdgeRequest{
            SourceId: sourceId, TargetId: targetId,
            Kind: "relates", Strength: 0,
        })
        writeJSON(w, http.StatusOK, map[string]any{
            "id": suggestion.ID, "type": "connect",
            "status": "accepted", "edgeId": edge.ID,
        })

    case "explore":
        // 前端本地处理，后端仅返回确认
        writeJSON(w, http.StatusOK, map[string]any{
            "id": suggestion.ID, "type": "explore",
            "status": "accepted",
        })

    default:
        writeError(w, http.StatusBadRequest, "unknown suggestion type")
    }
}
```

### 4.3 LLM 生成节点内容

**Prompt 策略：**
- 输入：`description`（规划建议描述）+ 湖内现有节点 title 列表（上下文）
- 输出：JSON `{ "title": "...", "content": "...", "tags": ["..."] }`
- LLM 不可用 fallback：`title = suggestion.Title`，`content = suggestion.Description`，`tags = []`

### 4.4 目标节点搜索逻辑（connect）

优先使用 `relatedNodeIds[1]` 作为目标节点；如果不存在，则根据以下策略搜索：
1. 在湖内搜索 title/description 关键词匹配度最高的节点
2. 排除源节点本身
3. 如果搜索结果为空，返回 `targetId = ""`，后端返回 400 错误

## 五、前端实现

### 5.1 数据流变更

**改前：**
```typescript
acceptPlanningSuggestion(id: string)
→ POST /api/graph/planning/:id
→ 后端根据 id 查询（TODO 空实现）
```

**改后：**
```typescript
acceptPlanningSuggestion(suggestion: PlanningSuggestion)
→ PUT /api/graph/planning/accept
   body: full PlanningSuggestion JSON
→ 后端根据 body 内容执行
```

### 5.2 PlanningPanel 采纳交互

```
用户点击 "一键采纳" 按钮
    ↓
按钮变为 loading 状态（disabled + spinner 文案："采纳中…"）
    ↓
调用 acceptPlanningSuggestion(suggestion)
    ↓ 成功
Toast: "已采纳" + 自动关闭该建议卡片 + 刷新图谱节点
    ↓ 失败
Toast: "采纳失败：{error message}"（按钮恢复可点击）
```

### 5.3 explore 类型特殊处理

`explore` 类型后端仅返回确认，前端在收到成功后自行：
- 在图谱中 highlight/focus `suggestion.relatedNodeIds[0]` 对应节点
- 缩放至该节点位置（参考 NodeDetailPanel 已有逻辑）

## 六、API 契约

### 6.1 请求

```
PUT /api/graph/planning/accept
Content-Type: application/json

{
  "id": "suggestion-uuid",
  "lake_id": "lake-id",
  "type": "add_node" | "connect" | "explore",
  "title": "建议标题",
  "description": "建议描述（LLM 用）",
  "priority": "high" | "medium" | "low",
  "related_node_ids": ["node-id-1", "node-id-2"]
}
```

### 6.2 响应

```
200 OK
{
  "id": "suggestion-uuid",
  "type": "add_node" | "connect" | "explore",
  "status": "accepted",
  "nodeId": "newly-created-node-id",    // 仅 add_node
  "edgeId": "newly-created-edge-id"    // 仅 connect
}

400 Bad Request
{
  "error": "target node not found" | "unknown suggestion type"
}
```

---

## 七、错误处理策略

| 场景 | 处理 |
|------|------|
| LLM 不可用 | fallback：title=description, content=description, tags=[] |
| 目标节点不存在（connect） | 返回 400 + "target node not found" |
| 边已存在（connect） | 幂等返回 existing edgeId，不报错 |
| explore 类型 | 始终返回 200，前端自行处理 |
